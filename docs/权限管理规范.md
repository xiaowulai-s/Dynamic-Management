## 1. 角色定义

系统采用三级角色模型，角色字段存储于 `users.role` 列，取值如下：

| 角色标识 | 显示名称 | 角色定位 |
|---------|---------|---------|
| `super_admin` | 超级管理员 | 系统最高权限，可管理所有用户与数据 |
| `admin` | 管理员 | 日常业务管理，可管理普通用户但不能管理超级管理员 |
| `user` | 普通用户 | 现场操作人员，只能查看、提交和编辑自己的日志 |

### 1.1 默认账号

| 用户名 | 角色 | 默认密码 | 说明 |
|--------|------|---------|------|
| `root` | super_admin | `root123` | 超级管理员，系统初始化创建 |
| `admin` | admin | `admin123` | 管理员 |

> 生产环境部署后请立即修改默认密码。

### 1.2 测试账号体系（v1.4 新增）

为避免开发测试时重置真实账号密码导致用户无法登录，系统提供专用测试账号：

| 用户名 | 角色 | 密码 | 用途 |
|--------|------|------|------|
| `admin_test` | admin | `123456` | 管理员场景开发验证 |
| `user_test` | user | `123456` | 普通用户场景开发验证 |

**测试账号规则**：

| 规则 | 说明 |
|------|------|
| 标识方式 | `users.is_test` 字段（Boolean，默认 false），测试账号设为 true |
| 可见性 | 仅 super_admin 在用户管理页可见；admin 角色查询时自动过滤 `is_test=true` |
| 使用场景 | 开发验证管理员/用户功能时，用 admin_test / user_test 登录 |
| 超管验证 | 用 root 账号验证，root 是真实超管账号 |
| 禁止事项 | **严禁重置真实账号（root/admin 及业务用户）密码**，如密码遗忘须向用户确认最新密码，不可擅自重置 |

**开发验证账号选择指引**：

| 验证场景 | 使用账号 |
|---------|---------|
| 超管功能（用户管理、删超管等） | `root` / `root123` |
| 管理员功能（审批、删设备等） | `admin_test` / `123456` |
| 普通用户功能（提交日志、查看） | `user_test` / `123456` |
| 菜单可见性对比 | 三个账号分别登录对比 |

## 4. 接口权限清单

所有受保护接口通过 `get_current_user` 依赖注入当前用户，并在接口内部校验角色。

### 4.1 认证接口（`/api/auth`）

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/auth/login` | 公开 | 登录 |
| POST | `/auth/register` | 公开 | 注册（仅创建 user 角色） |
| GET | `/auth/me` | 已认证 | 获取当前用户信息 |
| PUT | `/auth/change-password` | 已认证 | 修改自己的密码 |
| GET | `/auth/users` | admin / super_admin | 用户列表（admin 过滤 super_admin） |
| POST | `/auth/users` | admin / super_admin | 创建用户（admin 不能创建 super_admin） |
| PUT | `/auth/users/{id}` | admin / super_admin | 更新用户（admin 不能改 super_admin） |
| DELETE | `/auth/users/{id}` | admin / super_admin | 删除用户（admin 不能删 super_admin；不能删自己） |

### 4.2 设备接口（`/api/equipment`）

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/equipment` | 已认证 | 设备列表 |
| GET | `/equipment/{id}` | 已认证 | 设备详情 |
| POST | `/equipment` | admin / super_admin | 新增设备 |
| PUT | `/equipment/{id}` | admin / super_admin | 更新设备 |
| DELETE | `/equipment/{id}` | admin / super_admin | 删除设备（有关联日志时拒绝） |

### 4.3 日志接口（`/api/logs`）

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/logs` | 已认证 | 日志列表 |
| GET | `/logs/{id}` | 已认证 | 日志详情 |
| POST | `/logs` | 已认证 | 提交日志 |
| PUT | `/logs/{id}` | 创建者 / admin / super_admin | 更新日志（仅 pending/rejected 状态） |
| POST | `/logs/{id}/approve` | admin / super_admin | 审批日志 |
| DELETE | `/logs/{id}` | admin / super_admin | 删除日志 |

## 5. 前端权限控制实现

### 5.1 User Store（`src/stores/user.ts`）

```ts
// 计算属性
const isAdmin = computed(() => userInfo.value?.role === 'admin' || userInfo.value?.role === 'super_admin')
const isSuperAdmin = computed(() => userInfo.value?.role === 'super_admin')
const isNormalUser = computed(() => userInfo.value?.role === 'user')
```

### 5.2 菜单级权限（`Layout.vue`）

管理类菜单使用 `v-if="userStore.isAdmin"` 控制，普通用户不可见。

### 5.3 按钮级权限

**日志操作按钮**（`Logs.vue`）：
- 查看：无 v-if，所有角色可见
- 编辑：`v-if="userStore.isAdmin || row.operator_id === userStore.userInfo?.id"`
- 审批：`v-if="userStore.isAdmin && row.status === 'pending'"`
- 删除：`v-if="userStore.isAdmin"`

**设备操作按钮**（`Equipment.vue`）：
- 查看：无 v-if
- 新增/编辑/删除：`v-if="userStore.isAdmin"`

**用户操作按钮**（`UserManage.vue`）：
- 添加用户：`v-if="userStore.isAdmin"`
- 编辑用户：`v-if="userStore.isAdmin"`
- 删除用户：`canDelete(row)` — 不能删自己；admin 不能删 super_admin

## 6. 越权防护规则

后端在每个接口内部做角色校验，前端控制仅为体验优化，**不能作为安全边界**。关键防护规则：

1. **用户列表过滤**：admin 登录时，`GET /auth/users` 在 SQL 层面 `WHERE role != 'super_admin'`，即使绕过前端也无法获取超管信息。
2. **用户操作边界**：admin 对 super_admin 的增/改/删操作在后端返回 403。
3. **自删防护**：任何角色都不能删除自己，后端校验 `current_user.id == user_id` 返回 400。
4. **日志编辑状态**：后端校验 `log.status in ('pending', 'rejected')`，已通过日志任何角色都不可改。
5. **日志删除**：普通用户即使构造 DELETE 请求，后端返回 403。

## 7. 删除操作与数据完整性规则

删除操作除受角色权限控制外，还受数据完整性约束。本节定义各资源的删除策略（v1.2 新增）。

### 7.1 删除用户

| 场景 | 处理策略 |
|------|---------|
| 删除自己 | 拒绝，返回 400 "不能删除当前登录用户" |
| admin 删除 super_admin | 拒绝，返回 403 |
| 删除有关联数据的用户 | **置空所有外键引用后真实删除**，保留历史数据 |

被删除用户的历史数据保留，引用该用户的字段置空：
- `equipment.created_by` → NULL（设备创建人显示"未知"）
- `logs.operator_id` → NULL（日志操作人显示"未知"）
- `logs.approver_id` → NULL
- `feedbacks.user_id` → NULL（反馈提交人显示"未知"）
- `feedbacks.replied_by` → NULL
- `approval_configs.created_by` → NULL
- `system_configs.updated_by` → NULL
- `audit_logs.user_id` → NULL
- `notifications` → 级联删除（cascade delete-orphan）

> 数据库迁移：`equipment.created_by`、`logs.operator_id`、`feedbacks.user_id` 已由 `NOT NULL` 改为可空，模型层同步对齐。

### 7.2 删除设备

| 场景 | 处理策略 |
|------|---------|
| 设备无关联日志 | 直接删除 |
| 设备有关联日志，`force=false`（默认） | 拒绝，返回 400，提示日志数量 |
| 设备有关联日志，`force=true` | 级联删除所有关联日志（含具体日志表）后删除设备 |

前端 `Equipment.vue` 删除流程：
1. 点击删除 → 确认框"确定要删除设备吗？"
2. 调用 `DELETE /equipment/{id}`（不带 force）
3. 若返回"有关联日志"错误 → 弹二次确认"强制删除将一并清除日志，是否继续？"
4. 用户确认 → 调用 `DELETE /equipment/{id}?force=true`
5. 读取 `error.response.data.detail` 显示后端错误信息

### 7.3 删除客户

| 场景 | 处理策略 |
|------|---------|
| 客户无关联设备 | 直接删除 |
| 客户有关联设备 | 拒绝，返回 400，提示"请先转移或删除这些设备" |

> 客户与设备通过 `equipment.customer_name` 字符串字段关联（非外键），删除客户不会触发外键约束，但会产生孤儿引用，因此前置检查拒绝。

### 7.4 删除日志

日志删除采用级联删除策略，会同时删除基础日志表记录和对应的具体日志表记录（InstallationLog/RepairLog 等 8 张表）。仅 admin/super_admin 可删除。

## 10. 后续维护

### 10.1 权限规则维护

- 新增菜单项时，在 `Layout.vue` 添加 `v-if` 控制，并同步更新本文档第 3 节
- 新增接口时，在接口内部添加角色校验，并同步更新本文档第 4 节
- 角色或权限规则变更时，本文档与设计画布 HTML 权限矩阵页须同步更新
- 可视化权限矩阵页：`design-proposals/pages/权限矩阵.html`
- 每次权限相关变更后，建议按第 9 节流程做三账号回归验证
- 新增资源删除接口时，按第 7 节策略选择"置空引用/级联删除/拒绝并提示"，并同步更新第 7 节

### 10.2 部署与发布流程

部署、二次开发流程、故障排查详见 `docs/前端设计规范.md` 第 7 节，此处仅列权限相关的关键点：

| 操作 | 命令/脚本 | 权限相关注意事项 |
|------|----------|----------------|
| 启动全套服务 | 双击 `启动系统.bat` | 启动后用 root/admin/testuser 三账号登录验证菜单与按钮权限 |
| 改前端代码后发布 | 双击 `更新前端.bat` | 涉及权限的 UI（菜单 v-if、按钮 v-if）变更后，必须跑三账号回归验证 |
| 改后端权限接口 | `docker restart equipment-backend` | 改 `backend/app/routers/auth.py` 或 `logs.py` 后，用 curl 验证接口权限（参考第 9.2 节） |
| 改数据库角色字段 | `docker exec equipment-postgres psql ...` | 直接改 `users.role` 字段后，对应用户需重新登录刷新 token 中的 role |
| 紧急禁用某用户 | `UPDATE users SET is_active=false WHERE id=X` | 禁用后该用户下次请求会被 403 拦截，已签发的 token 在过期前仍有效 |
